Evidence for T8 — stale brand-kit refresh path
================================================

Scenario: ensureFreshBrandKitForWeeklyRun re-extracts when persisted kit is stale
Captured: 2026-05-06T18:30:00Z
Test path: tests/social-content-brand-kit-injection.test.ts
Test name: "ensureFreshBrandKitForWeeklyRun re-extracts when persisted kit is stale"

Method
------
1. Persist a tenant brand kit with `extracted_at: '2020-01-01T00:00:00.000Z'`
   under DATA_ROOT/generated/validated/<tenant_id>/brand-kit.json. The 7-day
   TTL inside extractAndSaveTenantBrandKit treats this as stale.
2. Inject a mocked `fetchImpl` that returns a small HTML page (with title,
   og:site_name, link rel=icon, link rel=stylesheet) for the brand URL plus a
   small CSS file for the stylesheet.
3. Call ensureFreshBrandKitForWeeklyRun({ doc, fetchImpl }) where
   doc.brand_kit is null and doc.inputs.brand_url points at the mocked host.
4. Assert that the function:
     - returned `{ refreshed: true }`
     - actually called fetch on https://brand.example
     - mutated doc.brand_kit so brand_name === 'Refreshed Brand'
     - persisted the refreshed brand kit to the DATA_ROOT JSON file

Observed behavior
-----------------
- result.refreshed === true (kit was re-extracted, not reused stale)
- fetchCalls includes 'https://brand.example' (HTML page) and
  'https://brand.example/assets/site.css' (stylesheet)
- doc.brand_kit.brand_name === 'Refreshed Brand'
- doc.brand_kit.extracted_at !== '2020-01-01T00:00:00.000Z' (refreshed timestamp)
- Persisted brand-kit.json on disk matches refreshed values

Failure-resistance proof points
-------------------------------
- ensureFreshBrandKitForWeeklyRun throws Error('needs_brand_kit:brand_url_missing')
  when doc.inputs.brand_url is empty.
- ensureFreshBrandKitForWeeklyRun throws Error('needs_brand_kit:<reason>') when
  extractAndSaveTenantBrandKit fails (network 5xx, parse failure, insufficient
  brand signals). The `needs_brand_kit:` prefix is a stable contract consumed
  by HermesMarketingPort.refreshBrandKitOrFail to surface a `needs_brand_kit`
  failure code on MarketingExecutionResult.error.code without crashing the
  port or the orchestrator.

Port-level wiring evidence
--------------------------
HermesMarketingPort.invoke() now calls refreshBrandKitOrFail BEFORE creating
the execution run record, so a stale kit triggers refresh; a refresh failure
returns kind='completed' / output.error.code='needs_brand_kit' instead of
fabricating a Hermes run that would be guaranteed to produce off-brand
generations.
